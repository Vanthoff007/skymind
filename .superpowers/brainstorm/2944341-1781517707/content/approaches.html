Three Approaches to Foundation-Model UAV Navigation

All three use AirSim + PX4 + ROS 2. The difference is where the foundation model sits in the loop.

A

VLM-as-Planner (Recommended)

Architecture: CLIP/LLaMA-Vision receives RGB-D frame → generates a language description of the scene → LLM decides next exploration frontier in natural language → PX4 waypoint controller executes.

Stack: CLIP or LLaVA for perception, GPT-4V or local LLaMA-3-Vision for planning, existing PX4 waypoint API for execution.

Pro: Cleanest separation of concerns. Each component is independently testable. Very adaptable to ground-robot baselines (NavGPT, VoroNav). Strong story for master's apps.

Con: API latency if using cloud VLM. Inference speed may limit real-time use.

B

CLIP Embeddings + Classical Planner

Architecture: CLIP encodes each camera frame into a semantic embedding → these embeddings replace raw pixels in a classical exploration planner (frontier-based or potential field) → PX4 executes.

Stack: CLIP for embeddings, custom frontier planner consuming semantic features, PX4 for control.

Pro: Faster inference (no LLM in the loop). Easier to evaluate — direct comparison to classical frontier exploration. More deterministic behavior.

Con: Less "foundation model" story — CLIP is doing feature extraction, not reasoning. Weaker novelty claim.

C

End-to-End Visuomotor Fine-Tuning

Architecture: Fine-tune a small VLM (e.g., OpenVLA or a LLaVA variant) on UAV navigation demonstrations collected in AirSim → model outputs velocity commands directly from image + task instruction.

Stack: AirSim for data collection, LoRA fine-tuning on LLaVA/OpenVLA, ROS 2 bridge to PX4.

Pro: Highest novelty. Directly analogous to RT-2 for drones. Most impressive if it works.

Con: Requires large demonstration dataset. Fine-tuning is expensive (GPU hours). Highest risk of not converging in project timeline. Hard to debug.

My recommendation: Option A — best balance of novelty, feasibility, and timeline for a master's application project.
